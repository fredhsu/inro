# Inro

An inbox for agent-generated **Documents**, with immutable **Revisions** and browser previews. It can run locally or as a hosted server on another trusted machine.

## Development

```bash
npm install
npm test
npm run build
npm run dev -- --port 4317
```

## CLI

Start the local server:

```bash
npm run dev -- serve
# or after build
node dist/cli/inro.js serve --host 127.0.0.1 --port 4317
```

The server creates a data directory at `~/.inro`, generates a persistent bearer token in `~/.inro/token`, and stores Documents/Revisions in SQLite.

Send a Markdown file:

```bash
node dist/cli/inro.js send ./note.md --server http://127.0.0.1:4317
```

Useful options:

- `serve --data-dir DIR --token TOKEN --host HOST --port PORT`
- `send --token TOKEN --title TITLE --document-key KEY --source-agent NAME --revision-summary TEXT`

Binding to a non-localhost host prints a warning because the browser UI and API are token-protected but exposed to the chosen network.

## Hosted server / agent skill setup

On the server machine, bind Inro to a reachable interface only on a trusted LAN, Tailscale network, SSH tunnel, or behind HTTPS:

```bash
inro serve --host 0.0.0.0 --port 4317 --token "$INRO_TOKEN"
```

On the machine where the AI agent runs, configure the bundled skill helper once:

```bash
node .agents/skills/inro-preview/scripts/setup-inro-skill.mjs \
  --server https://inro.example.com \
  --token "$INRO_TOKEN" \
  --source-agent pi-coding-agent
```

The setup helper writes `~/.inro/client-config.json` with restrictive permissions, verifies the server/token by default, and never prints the token. The send helper then reads that config automatically.

Preferred agent path:

1. **Setup helper** for hosted/remote configuration.
2. **Portable send helper** for reliable sends without requiring a global `inro` binary.
3. **CLI fast path** when `inro` is installed and explicit credentials are available.

```bash
node .agents/skills/inro-preview/scripts/send-inro-document.mjs ./note.md \
  --title "Project Notes" \
  --source-agent pi-coding-agent
```

Environment variables are also supported: `INRO_SERVER_URL`, `INRO_TOKEN`, and `INRO_SOURCE_AGENT`.

## Agent examples

HTML report:

```bash
cp .agents/skills/inro-preview/templates/report.html .scratch/weekly-report.html
node .agents/skills/inro-preview/scripts/send-inro-document.mjs .scratch/weekly-report.html \
  --title "Weekly Reliability Report" \
  --document-key "weekly-reliability-report" \
  --revision-summary "Initial weekly report"
```

Markdown notes:

```bash
node .agents/skills/inro-preview/scripts/send-inro-document.mjs .scratch/notes.md \
  --title "Research Notes" \
  --source-agent pi-coding-agent
```

Append a Revision to an existing Document:

```bash
node .agents/skills/inro-preview/scripts/send-inro-document.mjs .scratch/notes.md \
  --document-id DOCUMENT_ID \
  --revision-summary "Added final section"
```

Recurring reports should use a stable **Document Key** at creation time. Save the returned `documentId`; subsequent runs should append with `--document-id` so they advance the same Document:

```bash
node .agents/skills/inro-preview/scripts/send-inro-document.mjs .scratch/daily-digest.md \
  --title "Daily Digest" \
  --document-key "daily-digest" \
  --revision-summary "Initial daily digest"
```

Cron/background agent example:

```cron
15 7 * * * cd /home/fred/project && ./scripts/generate-digest.sh && node .agents/skills/inro-preview/scripts/send-inro-document.mjs .scratch/daily-digest.md --document-id "$(cat .scratch/daily-digest.document-id)" --source-agent cron-digest-agent --idempotency-key "daily-digest-$(date -I)" --revision-summary "Daily digest $(date -I)" >> .scratch/inro-cron.log 2>&1
```

## Provenance convention

Generated Documents should include a short provenance footer/metadata block unless the user asks for a pristine output. Include Source Agent, generated timestamp, non-secret input summary, Document Key, Revision Summary, and worktree/repo when useful. Do not include bearer tokens, private prompt text the user did not ask to preserve, hidden chain-of-thought, or environment dumps.

Markdown footer:

```markdown
---
Generated by pi-coding-agent for Inro. Generated: 2026-05-26T00:00:00Z. Inputs: issue triage summary. Document Key: daily-digest.
```

HTML footer:

```html
<footer data-inro-provenance data-source-agent="pi-coding-agent" data-generated-at="2026-05-26T00:00:00Z" data-document-key="daily-digest">
  Generated by pi-coding-agent for Inro on 2026-05-26T00:00:00Z.
</footer>
```

## Diagnostics

The inro-preview skill documents upcoming diagnostics features being added for agents:

- `scripts/inro-doctor.mjs` for config discovery, reachability, auth, base-path, and returned-URL checks.
- `send-inro-document.mjs --dry-run` to show the resolved server/format/title without sending.
- `send-inro-document.mjs --json` for machine-readable `documentId`, `revisionId`, `latestUrl`, and `revisionUrl`.

Example once available:

```bash
node .agents/skills/inro-preview/scripts/inro-doctor.mjs --server https://inro.example.com --json
node .agents/skills/inro-preview/scripts/send-inro-document.mjs ./note.md --dry-run --json
```

## Troubleshooting HTTP errors

- **Connection refused / `fetch failed` against localhost**: the local Inro server is probably not running. Start it or point `--server`/`INRO_SERVER_URL` at the hosted server.
- **Hosted unreachable (`ENOTFOUND`, timeout, 502/503)**: check DNS, VPN/Tailscale, SSH tunnel, firewall, and that Inro is bound to a reachable host/port.
- **401/403 Unauthorized**: token mismatch. Use the hosted server token, not an unrelated local `~/.inro/token`; re-run setup or set `INRO_TOKEN` explicitly.
- **401 only through a reverse proxy**: ensure the proxy forwards the `Authorization` header.
- **404 or proxy HTML for `/api/documents`**: base path mismatch. The external URL must route `/api/...` to Inro.
- **CORS error in a browser**: the helper/CLI use server-side Node calls, so CORS is not relevant there. Treat helper failures as network/auth/base-path issues.
- **Localhost token used for remote**: configure `~/.inro/client-config.json` with the remote URL and remote token.
- **Returned localhost URLs do not open from phone/laptop**: send with the externally reachable `--server` URL; if needed, replace the localhost origin with the hosted origin while keeping the `/d/...` path.
